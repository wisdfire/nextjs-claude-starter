# 데이터 잡 요구사항 문서(DATA-JOBS) 작성 표준

## 왜 별도 문서인가 (Why-first)

이 저장소는 **Next.js 웹서비스만** 소유한다. 크롤링·주기 API 호출·배치는 별도 모노레포 **[`wisdfire/jobhub-jobs`](https://github.com/wisdfire/jobhub-jobs)** 가 구현한다. 그런데 "무슨 데이터가 왜 필요한가"를 아는 쪽은 **제품(PRD)** 이다.

그래서 경계를 이렇게 나눈다.

| 누가 | 무엇을 소유하는가 |
| --- | --- |
| 이 저장소(웹) | **요구사항** — 무슨 데이터를, 왜, 어느 주기로, 어떤 필드로, 얼마나 신선하게 |
| `jobhub-jobs` | **구현** — 어떻게 가져와서 어떻게 넣을지 |

`docs/DATA-JOBS.md`는 그 경계의 **인계 문서**다. PRD 본문에는 수집 요구를 길게 쓰지 않고 이 문서로 링크만 건다.

## 언제 만드는가 (필수 트리거)

PRD의 데이터 확보 전략에 다음이 **하나라도** 있으면 이 문서를 **반드시** 만든다.

- 외부 사이트를 **주기적으로 크롤링**해야 하는 데이터
- 외부 **REST/공공 API를 주기 호출**해 적재해야 하는 데이터
- 그 밖의 **배치·크론성 데이터 갱신**

수집 요구가 하나도 없으면 문서를 만들지 않고, "수집 요구 없음"을 명시적으로 보고한다(빈 문서를 남기지 않는다).

## 🚫 여기에 쓰지 않는 것 (구현 계약 — jobhub-jobs 소관)

아래는 **모노레포가 결정**한다. 추측해서 적으면 잘못된 지시가 되므로 쓰지 않는다.

- `run()` 진입점·모듈 구조·파이썬 코드
- 이미지 선택(slim / browser), Dockerfile, 베이스 핀
- `jobs.yml` 등록·크론 표현식(UTC 변환)·러너 재시도 정책
- `on_conflict` 컬럼 구현·SQL·upsert 코드
- SSM 비밀값 등록 절차, CI/CD

요구사항 문서는 **"무엇이 같은 레코드인가"(유일성 정의)** 까지만 말하고, 그것을 `on_conflict`로 어떻게 구현할지는 말하지 않는다.

## 문서 구조

문서 하나에 잡이 여러 개일 수 있다. 상단에 공통 사항, 그 아래 **잡 단위 섹션**을 반복한다.

```markdown
# 데이터 잡 요구사항 (DATA-JOBS)

> 이 문서는 `wisdfire/jobhub-jobs` 모노레포가 읽어 구현하는 **요구사항 인계 문서**다.
> 구현 방식(run()·이미지·jobs.yml·upsert 구현)은 jobhub-jobs가 결정한다.
> 원천 PRD: `docs/PRD.md` §데이터 확보·유지보수 최소화 전략

## 개요
- 웹서비스: (한 줄)
- 잡 목록: (표 — 잡 키 / 종류(crawl·api) / 갱신 주기 / 적재 테이블)
- 공통 전제: 적재 대상은 Supabase(PostgreSQL). 웹앱은 적재된 결과를 **읽기만** 한다.

---

## 잡 1: <잡 키 제안>

### 왜 필요한가
- 이 데이터가 없으면 웹서비스의 어떤 화면·기능이 성립하지 않는가 (PRD 요구사항 참조)

### 데이터 소스
- 대상 URL / API 엔드포인트 (실제 확인한 것만. 추측 금지)
- **공식 API 존재 여부** — 있으면 크롤링 대신 API를 요구한다
- 인증 필요 여부: 필요한 **비밀값의 이름과 용도**만 적는다 (값·발급 절차는 쓰지 않음)
- **법적·정책 제약**: robots.txt 허용 여부 · 이용약관 · 라이선스 · 출처 표기 의무

### 필요 필드
| 필드 | 타입 | 필수 | 설명 | 없을 때 웹앱 영향 |
| --- | --- | --- | --- | --- |

### 갱신 주기 · 신선도 요구
- 요구 주기: (예: 1일 1회 / 분기 1회)
- **신선도(freshness) 허용치**: 데이터가 며칠까지 묵어도 화면이 유효한가
- 요구 주기는 **KST 기준 사람의 요구**로 적는다. UTC 크론 변환은 jobhub-jobs가 한다.

### 유일성 정의 (같은 레코드 판정)
- 무엇이 같으면 같은 레코드인가 — 자연키 조합 (예: `(site, season_id)`)
- ⚠️ 이건 **데이터 요구사항**이다. `on_conflict` 구현은 jobhub-jobs가 정한다.

### 적재 스키마 (제안)
- Supabase 테이블명 · 컬럼 · 타입 · 인덱스 요구(웹앱 조회 패턴 기준)
- **웹앱이 실제로 어떻게 조회하는지**를 함께 적어 인덱스 근거를 남긴다

### 웹앱 소비 방식
- 어느 라우트/화면이 이 테이블을 읽는가
- 🚨 **애드센스 제약(타협 불가)**: 수집 원본을 **그대로 게시하지 않는다.** 웹앱이 더할 **편집 부가가치**(요약·구조화·비교·해설·큐레이션)를 여기에 구체적으로 명시한다. 원본 나열은 복제·저가치 콘텐츠로 심사 거절 사유다 (`adsense-readiness` 스킬).

### 데이터 없음·실패 시 웹앱 동작
- 최초 적재 전 / 수집 실패로 데이터가 묵었을 때 화면이 무엇을 보여주는가
- ⚠️ **"준비 중" 플레이스홀더·빈 목록에 광고를 렌더하지 않는다**(애드센스 MUST)

### 미결 질문
- jobhub-jobs 쪽 판단이 필요한 항목
```

## 작성 규칙

- **근거 추적성**: 각 잡은 PRD의 어느 요구사항을 위한 것인지 참조를 남긴다. PRD에 없는 수집을 임의로 추가하지 않는다.
- **환각 금지**: 대상 URL·API 엔드포인트·필드는 **실제 확인한 것만** 적는다. 확인 못 했으면 "확인 필요"로 표시한다(그럴듯한 API 경로를 지어내지 않는다).
- **경계 침범 금지**: 위 "🚫 여기에 쓰지 않는 것"을 지킨다. 구현을 규정하려 들면 모노레포와 충돌한다.
- **잡 키 제안**: `jobhub-jobs`의 `sites/<사이트>` 키 컨벤션(소문자 snake_case)에 맞춰 **제안**하되, 최종 확정은 모노레포 몫임을 명시한다.
- **최소 수집 원칙**: 웹앱이 실제로 렌더링하는 필드만 요구한다. "언젠가 쓸지도 모르는" 필드는 넣지 않는다(수집 비용·차단 위험·유지보수 증가).

## 산출물

- `docs/DATA-JOBS.md` (정본). 재호출 시 전체 재작성이 아니라 변경된 잡 섹션만 최소 편집한다.
- PRD·ROADMAP에서는 이 문서를 **링크로 참조**하고 내용을 복제하지 않는다(중복은 곧 drift).
