# 파이프라인 모니터링 설계 방법론

모니터링의 목적은 "대시보드를 채우는 것"이 아니라 "문제를 늦지 않게 아는 것"이다. 지표를 분류하지 않으면 파이프라인은 초록불인데 데이터가 썩는 상황을 놓친다. **상태와 품질을 나눠서 관측하라.**

## Why-first: 왜 메트릭을 두 축으로 나누는가

파이프라인이 "돌았다"는 것과 데이터가 "맞다"는 것은 다르다. 실행은 성공했는데 값이 오염될 수 있고, 실패했는데 이전 데이터가 멀쩡할 수도 있다. 두 축을 분리해야 진짜 사고를 잡는다.

### 축 A — 파이프라인 상태 메트릭
- **볼륨(volume)**: 처리 행 수. 급감/급증은 소스 이상 신호.
- **지연(latency)**: 실행 소요·큐 대기 시간.
- **실패율(error rate)**: 실패 실행/레코드 비율, 재시도 횟수.

### 축 B — 데이터 품질 메트릭
- **신선도(freshness)**: `now − max(updated_at/loaded_at)`. schema가 제공한 타임스탬프 컬럼 기반.
- **품질 지표**: validation 설계의 완전성·정확성·일관성 산식을 **그대로 재사용**(중복 정의 금지).
- **드리프트(drift)**: 분포·스키마·볼륨의 시간적 변화.

## 임계값 산정 (Why: 임의 숫자는 알림 피로를 만든다)

- **과거 분포 기반**: 최근 N일의 평균·표준편차로 정상 범위를 정한다. 통계 근거 없이 임계값을 찍지 마라.
- **정적 vs 상대**: 절대 하한(예: 신선도 6시간 초과)과 상대 변화(전일 대비 ±30%)를 구분해 둘 다 쓴다.
- **초기값 + 튜닝 절차**: 과거 데이터가 없으면 보수적 초기값을 두고, 운영하며 조정하는 절차를 문서화한다.

## 알림 설계 (Why: 심각도 없는 알림은 무시된다)

- **심각도 분류**: critical(SLO 위반·데이터 중단) / warning(임계 근접) / info. 채널·대응자를 심각도별로 지정.
- **노이즈 억제**: 중복 알림 묶기, 히스테리시스(복구 확인 후 해제), 유지보수 윈도우 무시.
- 각 알림은 "무엇이·왜·다음 행동"을 담아 대응 가능하게(actionable).

## SLA/SLO·에러 버짓

- **SLO**: 신선도·가용성·정확성 목표를 수치로(예: "데이터 신선도 < 2시간 99%"). 
- **에러 버짓**: SLO 여유분. 소진 추세를 추적해 알림·대응 우선순위를 정한다.
- SLO → 알림 임계값 → 대시보드 지표가 한 방향으로 파생되게 정합을 유지한다.

## 데이터 드리프트 감지

- **분포 드리프트**: 수치 컬럼 평균/분위수, 범주 컬럼 분포를 기준 구간과 비교(예: PSI, KS 등 통계 지표).
- **스키마 드리프트**: 컬럼 추가/삭제/타입 변경 감지.
- **볼륨 드리프트**: 일별 행 수 추세 이탈.
- 각 드리프트에 대상·기준 구간·판정 임계·행동(알림/차단)을 명시한다.

## 산출물 형식 (`_workspace/04_monitoring_setup.md`)

- 메트릭 목록표: 지표 · 정의/산식 · 수집 소스 · 주기 · 축(상태/품질)
- 알림 규칙표: 조건·임계값 · 심각도 · 채널 · 대응 행동
- 대시보드 지표 구성
- SLA/SLO·에러 버짓 정의
- 드리프트 감지: 대상 · 기준 · 판정 · 행동

## 하지 말 것

- 상태 메트릭만 보고 품질을 안 보기(초록불 오염 사고).
- 통계 근거 없는 임계값 → 과거 분포 기반 산정.
- validation 품질 지표를 이름·산식 다르게 재정의 → 그대로 재사용.
- 신선도 컬럼 임의 가정 → schema 산출물에서 확인.
