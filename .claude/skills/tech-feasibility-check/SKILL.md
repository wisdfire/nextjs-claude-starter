# 기술 실현성 · 출처 정상성 검증

## 왜 검증하는가 (Why-first)

문서는 그럴듯해 보여도 (1) 참조한 API·버전이 이미 낡았거나 deprecated이고, (2) 인용한 URL이 죽었거나 아예 존재하지 않는(환각) 경우가 많다. 이 둘을 실제 도구로 확인하지 않으면, 구현 단계에서 "문서대로 했는데 안 되는" 비용이 발생한다. 기억에 의존한 단정을 금지하고, **도구로 확인한 사실만** 판정 근거로 삼는다.

## (a) 최신성 확인 절차

1. 문서가 참조하는 **라이브러리·프레임워크·SDK·API**를 목록화한다.
2. 각 항목을 **context7 MCP**로 조회해 현재 최신 버전·API 시그니처·권장 사용법을 확인한다.
   - 예: Next.js 16의 breaking change(`middleware.ts` → `proxy.ts`)처럼 훈련 데이터와 다른 변경을 잡는다.
3. context7로 부족하면 **공식 문서 / WebSearch**로 보강한다.
4. **deprecated·EOL·버전 불일치**를 발견하면 대체안과 함께 기록한다.

## (b) 출처 정상성 확인 절차

1. 문서의 **모든 인용 URL**을 목록화한다.
2. 각 URL을 **WebFetch로 실제 접근**한다.
   - **생존 확인**: 200 정상 응답인가, 404·차단·리다이렉트인가.
   - **내용 일치**: 페이지 실제 내용이 문서의 주장·인용과 일치하는가.
3. 문제를 분류한다.
   - **죽은 링크**: 접근 불가(404·도메인 소멸·차단).
   - **환각 출처**: URL·문서·저자가 애초에 존재하지 않음.
   - **내용 불일치**: 살아 있으나 주장과 다른 내용.
4. 일시 실패(네트워크)는 재시도로 구분하고, "접근 불가"와 "환각"을 혼동하지 않는다.

## 판정·심각도 등급화

- **Blocker**: 스택으로 구현 불가, 존재하지 않는 API, 환각 출처, 핵심 근거의 죽은 링크.
- **Warning**: deprecated API, 버전 위험, 내용 부분 불일치.
- **Info**: 개선 제안, 더 나은 대체 API.

각 문제에 **구체적 보정안**(대체 API·삭제·표현 수정)을 붙여 작성자가 바로 반영할 수 있게 한다.

## 원칙

- **추측 금지**: API·버전을 기억으로 단정하지 않는다. 도구 확인 결과만 쓴다.
- **불확실은 표기**: 확인 불가 항목은 "수동 확인 필요"로 남기고 임의 판정하지 않는다.
- **환각과 접근 불가 구분**: 존재하지 않는 것과 일시적으로 못 여는 것을 다르게 기록한다.
