# ETL 로직 설계 방법론

ETL의 목표는 "한 번 맞게 도는 것"이 아니라 "몇 번을 돌려도 같은 결과가 나오는 것"이다. 파이프라인은 반드시 재실행된다(실패·지연·백필). **재실행 안전성부터 설계하라.**

## Why-first: 왜 멱등성이 먼저인가

파이프라인은 네트워크 오류·타임아웃·부분 실패로 중간에 죽고 재시도된다. 멱등하지 않으면 재시도가 중복 적재·이중 집계를 낳는다. 멱등성을 먼저 보장하면 나머지(재시도·백필)가 안전한 반복이 된다.

- **Upsert(ON CONFLICT)** 로 적재하라 — INSERT 후 실패해도 재실행이 덮어쓴다.
- Upsert 키는 **schema 설계의 PK/유니크 키를 그대로** 쓴다(새 키 발명 금지).
- 집계·누적은 "델타 더하기"가 아니라 "구간 재계산"으로 설계해 재실행이 중복되지 않게 한다.

## 증분 적재 (Why: 전량은 비싸고 느리다)

전량 재적재는 소스·DB에 부담이고 신선도를 떨어뜨린다. 증분이 기본이다.

- **워터마크 기반**: `updated_at`/시퀀스/로그 오프셋을 저장해 "마지막 처리 지점 이후"만 추출한다.
- 워터마크는 **처리 완료 후** 커밋한다(처리 전 갱신하면 유실). 안전을 위해 약간 겹치게(overlap) 재추출하고 멱등성으로 중복을 흡수한다.
- **late/out-of-order 데이터**: 소스가 지연 도착·순서 뒤바뀜을 허용하면 이벤트 시각 기준 윈도우로 재처리한다. 처리 시각만 믿지 마라.
- CDC(변경 데이터 캡처)가 가능하면 삭제(delete)까지 반영하도록 tombstone 처리를 명시한다.

## Upsert 키 전략

- 자연키가 안정적이고 유니크면 그것을 충돌 키로. 아니면 대리키 + 유니크 제약.
- 복합 유니크(예: source_id + business_date)가 실질 식별자면 그것을 ON CONFLICT 대상으로.
- 충돌 시 갱신 컬럼을 명시(전체 덮어쓰기 vs 특정 컬럼만). `updated_at`은 항상 갱신.

## 배치 vs 스트리밍 (Why: 지연 요구가 결정한다)

- **배치**: 분/시간 단위 지연 허용, 대량 처리 효율, 단순함. 대부분의 분석 적재는 배치로 충분.
- **스트리밍**: 초 단위 신선도 필요, 이벤트 반응형. 운영 복잡도·비용 증가.
- 애매하면 배치로 시작하고 스트리밍 승격 조건을 문서화하라(오버엔지니어링 금지).

## 에러 재처리·백필

- **분류**: 재시도 가능(일시적 네트워크·락) vs 불가(스키마 불일치·검증 실패). 가능만 재시도한다.
- **재시도**: 지수 백오프 + 상한. 무한 재시도 금지.
- **격리(DLQ/quarantine)**: 반복 실패 레코드는 격리 저장소로 보내 파이프라인을 막지 않는다. validation 설계의 quarantine 흐름과 맞춘다.
- **백필**: 특정 기간을 안전하게 재적재하는 절차를 명시한다. 멱등성이 있으므로 해당 구간 재실행 = 정정. 파티션/구간 단위로 재계산한다.
- **적재 순서**: FK 의존을 지켜 부모→자식 순으로. schema의 관계도를 참조한다.

## 테스트·검증 전략 (필수 — 설계물에 명시)

설계는 "이 로직을 어떻게 테스트할지"까지 못박아야 한다. Why: 테스트 전략을 설계 단계에서 고정해야 구현팀이 회귀를 막고 데이터 품질을 보장한다. 아래를 **필수 섹션**으로 산출물에 포함한다.

- **Transform 단위테스트**: 대표 fixture 입력 → 기대 출력(정제·매핑·타입 변환·집계 결과)을 케이스로 정의한다. 정상·경계·이상 입력을 모두 커버한다.
- **멱등성 테스트**: 같은 배치를 **2회(이상) 실행**해도 대상 테이블의 행 수·집계가 변하지 않고 **중복이 0**임을 검증하는 케이스를 명시한다. Upsert 키가 실제로 충돌을 흡수하는지 확인한다.
- **증분·백필 경계 케이스**: 워터마크 경계(같은 타임스탬프 동시 도착), late/out-of-order 도착, overlap 재추출 중복 흡수, 특정 구간 백필이 기존 값을 **정정만** 하는지(중복 생성 없음)를 케이스로 정의한다.
- **테스트 도구 명시**: 구현 스택에 맞춰 어떤 도구로 돌릴지 못박는다(파이썬 구현이면 **pytest** + 고정 fixture, TypeScript/Next.js 구현이면 **Vitest**). 설계물은 도구명과 fixture 위치·실행 방식을 지정한다.

## 하지 말 것

- schema와 다른 필드명·키 사용 → 불일치 시 lead 통해 조율.
- 처리 전 워터마크 커밋(데이터 유실).
- 근거 없는 스트리밍 선택(불필요한 복잡도).
- 테스트 전략 없는 적재 로직 설계(멱등성·경계 케이스 검증 방법 누락).
