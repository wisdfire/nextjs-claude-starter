# 와이어프레임 & 디자인 시스템 설계

요구사항을 이후 엔지니어가 그대로 구현할 수 있는 **설계 명세**로 번역한다. 이 단계의 정확도가 파이프라인 전체 품질을 좌우한다. 특히 라우트 매핑은 이후 QA의 링크 검증 기준(SSOT)이 되므로 틀리면 안 된다.

## 왜 이 단계가 중요한가

구현을 먼저 하고 설계를 나중에 맞추면, 화면 누락·라우트 불일치·컴포넌트 재작업이 연쇄로 발생한다. 설계를 명세로 고정하면 프론트/백엔드가 병렬로 착수할 수 있고, QA가 대조할 기준이 생긴다.

## 절차

### 1. 정보구조(IA) 도출

- 요구사항에서 **명사(엔티티)**와 **동사(행위)**를 뽑아 화면 목록을 만든다.
- 화면 간 이동 흐름(내비게이션)을 그린다: 진입점 → 주요 흐름 → 종료/전환.
- 요구사항에 없는 화면을 지어내지 않는다. 추론이 필요하면 "가정:"으로 명시한다.

### 2. 화면 ↔ 라우트 매핑 (가장 중요)

각 화면을 Next.js 16 App Router 경로로 매핑하여 표로 만든다. **이 표가 SSOT다.**

- route group `(group)`은 **URL에서 제거**된다: `app/(marketing)/about/page.tsx` → URL `/about`.
- 동적 세그먼트 `[param]`은 URL 파라미터가 된다: `app/posts/[id]/page.tsx` → `/posts/:id`.
- 인증 영역은 route group으로 묶는 것을 권장: `app/(dashboard)/create/page.tsx` → URL `/create`가 아니라, 실제 URL이 무엇인지 **명확히** 적는다. 접두사 유무를 헷갈리면 이후 링크가 404가 된다.

표 형식:

```markdown
| 화면 | 라우트(URL) | page 파일 경로 | 접근 권한 |
|------|-------------|----------------|-----------|
| 대시보드 | /dashboard | app/(app)/dashboard/page.tsx | 로그인 |
| 글 작성 | /dashboard/create | app/(app)/dashboard/create/page.tsx | 로그인 |
| 글 상세 | /posts/[id] | app/posts/[id]/page.tsx | 공개 |
```

### 3. 와이어프레임 (저충실도)

각 화면의 레이아웃을 텍스트/구획으로 표현한다. 픽셀 완벽함이 목표가 아니라 **구성 요소와 배치, 상태(빈/로딩/에러)**를 드러내는 것이 목표다.

- 영역(헤더/사이드/본문/푸터)과 각 영역의 요소를 나열한다.
- 주요 상호작용(클릭 시 이동, 제출 시 동작)과 이동 목적지(라우트)를 적는다 → 프론트가 링크를 정확히 걸 근거.
- 빈 상태·로딩·에러 화면도 명세한다(누락되면 구현에서 빠진다).

### 4. 디자인 토큰 방침 (shadcn 토큰 수정 금지)

- **`app/globals.css`의 `:root`·`.dark` CSS 변수와 `@theme inline` 블록을 수정하지 않는다.** 이 프로젝트의 확정 규칙이다.
- 색·간격·타이포는 shadcn 기본 토큰과 Tailwind v4 유틸리티로 표현한다. 커스텀이 꼭 필요하면 "토큰 변경 없이 유틸리티 조합으로 달성" 방향을 제시한다.
- 시각 방향(무드/팔레트/폰트 페어링/레이아웃 스타일)은 `ui-ux-pro-max` 스킬로 결정하고, 미감·타이포 디테일은 `frontend-design` 스킬을 참고한다.

### 5. 컴포넌트 명세

- 화면별로 필요한 컴포넌트를 나열한다.
- 각 컴포넌트에 대해 **먼저 shadcn 카탈로그에서 찾는다**: `npx shadcn@latest add <name>`로 추가할 후보를 지정한다(button, card, dialog, form, table, input 등).
- shadcn은 base-ui 기반이다. 합성이 필요한 컴포넌트는 Radix `asChild`가 아니라 base-ui **`render` prop**을 전제로 명세한다.
- shadcn에 없는 컴포넌트만 "직접 구현" 항목으로 분리한다.

## 산출물

`_workspace/01_design_spec.md` 하나에 위 1~5를 순서대로 담는다. 와이어프레임이 길면 `_workspace/01_wireframes.md`로 분리하되, 라우트 표는 반드시 `01_design_spec.md`에 둔다.

## 하지 말 것

- 라우트 표에서 route group 접두사를 임의로 붙이거나 빼지 않는다(실제 파일 경로 기준으로 URL을 유도).
- 요구사항에 없는 화면·기능을 추가하지 않는다.
- shadcn 디자인 토큰(`:root`/`.dark`/`@theme inline`)을 바꾸는 명세를 내지 않는다.
- 데이터가 필요한 화면인데 어떤 데이터가 필요한지 적지 않은 채 넘기지 않는다(백엔드가 API를 설계할 근거가 사라진다).


## 애드센스 요건 (설계 단계 필수 · 최우선)

> **`adsense-readiness` 스킬이 단일 진실 공급원**이다. 반드시 함께 로드하라. 애드센스 심사 통과는 타협 불가 목표이며, 요건은 **설계 단계에서** 반영해야 한다(사후에 붙이면 라우트·레이아웃을 다시 짜야 한다).

와이어프레임·IA·라우트 표에 아래를 반영한다:

- **정책 페이지 라우트 필수 포함**: `app/privacy`(개인정보처리방침 — **유일한 공식 필수 페이지**)를 반드시 넣고, 모든 화면의 **푸터 등 전역 내비게이션에서 도달 가능**하게 명세한다. `about`·`contact`·`terms`는 UX·내비게이션 요건 충족용으로 권장(정책상 필수는 아니므로 "애드센스 필수"라고 쓰지 말 것).
- **광고 허용 화이트리스트**: 광고를 렌더할 라우트를 명시한다. **로그인·에러(404/500)·Thank you·빈 목록·스켈레톤 화면에는 광고를 두지 않는다**(저가치 화면 광고 = 거절 사유).
- **콘텐츠 오리지널리티 레이아웃**: 수집 데이터를 쓰는 화면은 **원본 나열이 아니라** 요약·비교·집계·큐레이션·해설이 드러나는 구조로 설계한다(복제·저가치 콘텐츠 = 대표 거절 사유).
- **플레이스홀더 금지**: "준비 중" 화면을 설계에 넣지 않는다.
- 광고가 콘텐츠를 압도하거나 내비게이션·버튼과 혼동되는 배치를 설계하지 않는다.
