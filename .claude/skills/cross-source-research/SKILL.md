# 교차 출처 리서치 방법론

## 왜 이렇게 하는가 (Why-first)

단일 출처·단일 축의 리서치는 편향된다. 웹(시장·현황)·학술(근거의 질)·커뮤니티(실사용자 수요)를 **독립적으로** 수집한 뒤 교차 검증하면, 한 축의 오류나 편향을 다른 축이 잡아낸다. 목적은 "많이 모으기"가 아니라 **믿을 수 있는 합의와 아직 불확실한 이견을 구분**하는 것이다.

## 3축 수집 원칙

각 축은 서로 대기하지 말고 **병렬·독립**으로 수집한다.

- **웹 축**: 시장 규모·경쟁 서비스·공식 문서·최신 뉴스·가격. `WebSearch`로 후보를 넓히고 `WebFetch`로 본문을 확인.
- **학술 축**: 논문·기술 표준·공식 벤치마크·기관 통계. 방법론·표본·한계를 함께 기록. 1차 출처를 우선.
- **커뮤니티 축**: 레딧·HN·포럼·SNS·리뷰의 페인포인트·수요 신호·감정. 신호 강도(빈도·업보트·규모)를 병기.

라이브러리·프레임워크 스펙이 관련되면 **context7 MCP**로 최신 버전·API를 교차한다.

## 출처 신뢰도 등급 (반드시 표기)

모든 주장 뒤에 등급을 단다. 등급 없는 단정은 쓰지 않는다.

- **A**: 피어리뷰 논문·공식 표준·1차 통계·공식 문서.
- **B**: 프리프린트·업계 표준 리포트·공신력 있는 매체.
- **C**: 블로그·2차 인용·개별 커뮤니티 일화.

표기 형식: `[출처: 제목 — URL — 접근일 — 등급]`

## 교차 검증 방법론

1. 세 축의 발견을 같은 주제끼리 나란히 놓는다.
2. **2축 이상 일치 → 합의**, **갈림 → 이견**으로 분류한다.
3. **상충 시 삭제하지 말고 양쪽을 출처와 함께 병기**한다. 판단은 신뢰도 가중(A·다축 합의 > B·단축 > C·일화)으로 우선순위만 매긴다.
4. 모든 축이 같은 단일 출처를 재인용한 "가짜 합의"를 경계한다 — 독립성이 없으면 합의로 승격하지 않는다.
5. 결과를 `검증된 사실 / 유력 가설 / 미해결 쟁점`으로 등급화한다.

## 환각 방지 (필수)

- WebFetch로 실제 확인하지 않은 URL·수치·인용·논문을 **지어내지 않는다**. 확인 실패는 "미확인"으로 명시한다.
- 정성 신호(커뮤니티)를 정량 사실처럼 과일반화하지 않는다. 대표성·편향을 함께 적는다.

## 산출물 규약

- 각 축은 `_workspace/02_{web|academic|community}_research.md`에 저장.
- 교차 검증 결과는 `_workspace/03_validated_findings.md`에 `합의 / 유력 가설 / 이견(양쪽 출처) / 미해결 쟁점`으로 구조화.
- 모든 발견은 원 출처로 추적 가능해야 한다.
