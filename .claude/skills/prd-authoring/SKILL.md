# PRD 작성 표준

## 왜 이렇게 쓰는가 (Why-first)

PRD는 "무엇을 왜 만드는가"를 근거와 함께 확정해 개발팀의 판단 비용을 없애는 문서다. 이 하네스의 PRD는 세 가지를 **반드시 요구사항으로 명문화**한다는 점에서 일반 PRD와 다르다: (1) **애드센스 심사 통과(타협 불가·최우선)**, (2) 유지보수 비용 최소화, (3) 객관적 완성도 지표. 이 세 가지가 빠지면 나중에 수익화 실패·운영 과부하·품질 미달로 되돌아온다.

## 표준 구조

1. **개요·문제 정의** — 검증된 페인포인트에 근거한 문제.
2. **타깃 사용자** — 수요 신호에 근거한 사용자군.
3. **목표·성공 지표** — 정량 목표(측정 가능).
4. **기능 요구사항** — 우선순위(MoSCoW: Must/Should/Could/Won't)와 함께.
5. **비기능 요구사항** — 성능·보안·접근성·SEO.
6. **애드센스 필수 요구사항(MUST 10항)** — 아래 (a).
7. **데이터 확보·유지보수 최소화 전략** — 아래 (b).
8. **완성도 검증 지표** — 아래 (c).
9. **품질·테스트 전략** — 아래 (d).
10. **범위 밖·리스크** — 안 하는 것, 미검증 가설.
11. **근거 출처** — 각 요구사항의 소급 근거.

## (a) 애드센스 심사 통과 요건 (최우선 · 요구사항으로 기입)

**`adsense-readiness` 스킬이 단일 진실 공급원**이다. 반드시 로드해 그 **공식 MUST 10항 전부**를 PRD에 검증 가능한 완료 기준과 함께 명문화한다. 애드센스 요건을 기억·추측으로 쓰지 않는다.

- [ ] **개인정보처리방침** — **유일한 공식 필수 페이지**. 구글 지정 문구(제3자 쿠키 광고·구글 광고 쿠키·opt-out 안내) 포함.
- [ ] **인증 CMP(IAB TCF)** — EEA/UK/스위스 트래픽이 있으면 필수.
- [ ] **HTTPS 강제 · 공개 접근 · `Mediapartners-Google` 크롤러 차단 금지**.
- [ ] **오리지널 콘텐츠** — 수집 데이터를 **원본 그대로 게시 금지**. 요약·구조화·비교·해설 등 **편집 부가가치** 필수(복제·저가치 = 대표 거절 사유).
- [ ] **자동생성/AI 콘텐츠 대량 게시 금지**(사람 검토·큐레이션 없는 것).
- [ ] **저가치 화면(로그인·404/500·빈 목록·스켈레톤)에 광고 슬롯 렌더 금지**, 플레이스홀더("준비 중") 화면 금지.
- [ ] **광고:콘텐츠 비율**·라벨·배치 규칙 준수(오클릭 유도 금지).
- [ ] **내비게이션 정상 동작 · 링크 404 없음**(Site behavior 요건).

⚠️ **MUST/SHOULD를 뒤섞지 말 것.** 공식 최소 분량·글 수 기준은 **존재하지 않는다** — "300자 이상", "글 20개", "About·Contact는 애드센스 필수" 같은 **근거 없는 수치를 지어내지 않는다**(About·Contact·이용약관은 UX상 권장일 뿐 정책상 필수 아님). "100% 통과 보장"이라고 쓰지 말고, **MUST 전항 충족 + 저가치 콘텐츠 리스크 제거**라는 검증 가능한 게이트로 표현한다.

## (b) 유지보수 최소화 원칙 (요구사항으로 기입)

수동 운영을 설계에서 배제한다.

- **자동 수집 요구사항**: 수집 대상·주기·필요 필드·신선도(freshness)를 **요약 수준으로만** 적고, 상세는 별도 문서로 뺀다(아래).
- 🚨 **수집 요구가 있으면 `docs/DATA-JOBS.md`로 분리(필수)**: PRD 본문에 수집 상세를 늘어놓지 말고 **`docs/DATA-JOBS.md`(모노레포 인계 문서)를 링크로 참조**한다. 그 문서는 `data-jobs-spec` 스킬 기준으로 data-jobs-spec-author가 작성한다. **PRD와 내용을 복제하지 않는다** — 두 곳에 같은 수집 스펙이 있으면 반드시 drift한다.
- ⚠️ **수집 구현은 이 저장소 범위 밖**: 크롤러·Python 배치·ETL은 별도 모노레포 **`wisdfire/jobhub-jobs`** 소관이다. PRD에는 수집 **요구사항**만 적고 **구현 방식(Celery·Playwright·크론 표현식 등)을 이 저장소의 작업으로 기술하지 않는다.** 웹앱의 책임은 "Supabase에 적재된 데이터를 읽어 렌더링"까지다.
- **실패 복원력**: 재시도·백오프·실패 알림을 요구사항에 포함. 운영자 개입 지점을 최소화하고, 자동화 불가 항목은 명시적으로 표시.
- **배포 설정(config-as-code)**: 배포·환경변수·크론을 수동 대시보드 조작이 아니라 **`vercel.ts` 코드로 관리**하도록 요구사항에 기입한다(재현성·drift 방지). 상세 규약은 `web-deploy-config` 스킬 참조.

## (c) 객관적 완성도 지표 (요구사항으로 기입)

주관적 "완성"이 아니라 측정 가능한 기준으로 완료를 정의한다.

- **Lighthouse 4개 카테고리 목표 점수**: Performance / Accessibility / Best Practices / SEO — 각 **90+**(프로젝트 기준에 맞게 조정 가능하되 수치 명시).
- **Core Web Vitals 임계값**, **접근성 위반 0건**, **단위 테스트 통과·커버리지** 등을 완료 기준(acceptance)으로 연결.
- **테스트 통과율을 완성도 지표에 포함**: Lighthouse 등 객관 지표와 짝으로, 아래 (d)의 **테스트 통과율(단위/통합/E2E)과 커버리지 목표치**를 완료 기준으로 명시한다.

## (d) 품질·테스트 전략 (요구사항으로 기입)

"동작한다"를 주관이 아니라 통과하는 테스트로 증명하도록, 요구사항 단계에서 테스트 범위를 확정한다.

- **테스트 범위 정의**: **단위(unit)·통합(integration)·E2E** 각 계층에서 무엇을 검증할지 명시한다(핵심 비즈니스 로직·API 계약·주요 사용자 흐름).
- **커버리지 기대치**: 핵심 모듈의 최소 커버리지 목표(예: 라인/브랜치 %)를 수치로 기입한다. 프로젝트 기준에 맞게 조정하되 반드시 수치를 남긴다.
- **CI 게이트**: **테스트가 통과해야만 배포**되도록 CI 게이트를 요구사항으로 명문화한다(테스트 실패 시 배포 차단).
- **완성도 연결**: 위 (c)의 완성도 지표에 테스트 통과율을 포함해, Lighthouse 점수와 테스트 통과가 함께 충족돼야 완료로 판정한다.

## 작성 규칙

- **시드 PRD는 가설로 취급**: 이 프로젝트의 PRD는 보통 `../auto-prd-vault/03_PRD_Docs/PRD-*.md`에서 파생된다. 시드의 주장·수치·출처는 **미검증 가설**이다. 검증된 발견으로 뒷받침되지 않은 시드 내용을 **그대로 복사하지 않는다**(근거가 없으면 "가설"로 표기하거나 뺀다).
- **근거 추적성**: 모든 요구사항은 `_workspace/03_validated_findings.md`의 검증된 발견으로 소급 가능해야 한다. 근거 없는 기능은 넣지 않는다.
- **환각 금지**: 미검증 시장 수치·주장을 단정으로 쓰지 않는다. 가설은 "가설"로 표기.
- **스택 정합**: Next.js 16 · Supabase · Drizzle · Vercel 전제. 벗어나는 요구는 표시해 tech-verifier 검증 대상으로 남긴다. 데이터 수집 구현은 모노레포 `jobhub-jobs` 소관이므로 이 저장소의 요구사항으로 쓰지 않는다.
- **완료 판정**: (a)(b)(c)(d)가 모두 요구사항으로 반영되고 각기 검증 가능한 완료 기준이 붙어야 PRD를 완료로 본다. **(a)의 애드센스 MUST는 전항 충족이 필수**이며, (b)에는 배포 config-as-code 원칙이, (d)에는 테스트 전략·CI 게이트가 포함돼야 한다.
