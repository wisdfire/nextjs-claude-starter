# 데이터 검증 규칙 체계

검증은 "나쁜 데이터를 어디서 잡는가"의 문제다. 계층을 나누지 않으면 어떤 규칙은 중복되고 어떤 결함은 통째로 빠져나간다. **계층으로 규칙을 배치하라.**

## Why-first: 왜 4계층인가

결함은 서로 다른 범위에서 드러난다 — 값 하나(타입), 레코드 하나(필드 간 모순), 데이터셋 전체(유니크·분포), 도메인 규칙(비즈니스 불변식). 한 층에서 다 잡으려 하면 놓친다. 층마다 잡을 것을 명확히 나눠라.

### 1) 스키마 레벨 (값 단위)
개별 값이 형식에 맞는가. **가능하면 DB 제약으로 강제**하라 — NOT NULL, 타입, CHECK(범위/패턴), UNIQUE. schema 설계의 제약을 그대로 참조하고 이중 정의하지 않는다. DB가 막는 결함은 애플리케이션 검증보다 강하다.

### 2) 레코드 레벨 (행 단위)
한 행 안의 필드 간 관계. 예: `end_date >= start_date`, `상태=완료면 완료시각 NOT NULL`. 단일 컬럼 제약으로 못 잡는 교차 필드 규칙을 여기서 정의한다.

### 3) 데이터셋 레벨 (집합 단위)
행들의 관계·분포. 예: PK 유니크성, 참조 무결성(FK 대상 존재), 행 수 급변 감지, 카테고리 분포·널 비율 임계. 배치 단위로 판정한다.

### 4) 비즈니스 레벨 (도메인 규칙)
도메인 불변식. 예: "일별 매출 합 = 거래 합", "환불액 ≤ 결제액". 모호하면 지어내지 말고 요구사항을 확인하라.

## 실패 처리: reject / quarantine / flag

규칙마다 실패 시 행동을 **반드시 하나 지정**하라. Why: 모두 reject하면 정상 데이터까지 막히고, 모두 통과시키면 오염된다.

- **reject**: 적재 거부. 무결성 치명(PK 중복, FK 위반, 필수 누락)에 사용. blocking.
- **quarantine**: 격리 저장소로 보내 파이프라인은 계속 흐르게. 검토·정정 후 재처리. 대량이거나 정정 가능성이 있는 결함에.
- **flag**: 적재하되 품질 플래그 컬럼으로 표시. 소비 측이 필터링하도록. 경미한 이상값·의심 데이터에.

quarantine 저장소는 **원본 + 실패 사유 + 유입 시각**을 보존해 재처리 가능하게 설계한다. ETL의 격리/재처리 흐름과 정합을 맞춘다.

## 품질 지표 정의 (측정 가능하게)

각 지표는 산식·대상 컬럼·임계값을 명시해 monitoring이 메트릭으로 연결하게 한다.

- **완전성(completeness)**: 필수 컬럼 널 비율 = 널 행 수 / 전체.
- **정확성(accuracy)**: 참조/기대값 대비 일치율(예: 코드값이 마스터에 존재).
- **일관성(consistency)**: 교차 필드·교차 테이블 규칙 위반율.
- **적시성(timeliness)**: 데이터 신선도 지연 = now − max(updated_at). 임계 초과 비율.

## 검증 규칙은 런타임 + 테스트 양쪽으로 (필수)

각 규칙은 "런타임에 어떻게 막는가"만이 아니라 "그 규칙이 실제로 동작함을 어떻게 증명하는가"까지 설계물에 담는다. Why: 규칙을 코드로 옮긴 뒤 회귀로 조용히 깨지면 오염 데이터가 통과한다. 규칙마다 대응 테스트 케이스를 페어링해 방어를 고정한다.

- **규칙 ↔ 테스트 페어링**: 계층별 규칙표의 각 행에 **대응 테스트 케이스**를 명시한다 — 위반 샘플 입력이 규칙에 걸려 지정된 실패 처리(**reject/quarantine/flag**)로 정확히 분기하는지 검증.
- **양성·음성 케이스**: 정상 샘플은 통과(적재)하고, 위반 샘플만 reject/quarantine/flag 되는지 양쪽을 케이스로 정의한다(과잉 차단·미차단 동시 방지).
- **품질 지표 테스트**: 각 품질 지표(완전성·정확성·일관성·적시성)가 임계값 경계에서 올바르게 판정/알림을 트리거하는 경계 케이스를 포함한다.
- **테스트 도구 명시**: 구현 스택에 맞춰(파이썬 구현이면 pytest, TypeScript면 Vitest) 검증 테스트를 어떤 도구로 돌릴지 지정한다. ETL의 테스트 전략과 도구·fixture 규약을 맞춘다.

## 산출물 형식 (`_workspace/03_validation_rules.md`)

- 계층별 규칙표: 대상 컬럼 · 조건 · 실패 처리(reject/quarantine/flag) · blocking 여부 · **대응 테스트 케이스**(위반 샘플→기대 분기)
- 품질 지표 정의표: 지표 · 산식 · 대상 · 임계값 · **경계 판정 테스트 케이스**
- quarantine 저장소 스키마·재검토/재처리 절차
- 규칙 우선순위(무결성 치명 규칙 우선)
- 검증 테스트에 사용할 도구(pytest/Vitest 등)

## 하지 말 것

- schema 제약과 규칙을 어긋나게 이중 정의.
- 실패 처리 미지정 규칙(무엇을 할지 모호).
- 근거 없이 과도한 reject → quarantine/flag 대안 검토.
- 모호한 비즈니스 규칙 지어내기 → 요구사항 확인.
- 대응 테스트 케이스 없는 검증 규칙(위반 샘플이 실제로 걸리는지 증명 누락).
