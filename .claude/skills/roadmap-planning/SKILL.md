# 로드맵 계획 방법론

## 왜 이렇게 하는가 (Why-first)

PRD의 요구사항은 "무엇을"까지만 말한다. 로드맵은 그것을 **순서·의존성·검증 기준이 붙은 실행 단위**로 바꿔야 개발이 멈추지 않는다. 체크박스로 진행을 추적하고, Task마다 객관적 완료 기준을 걸면 "얼마나 됐는지"와 "정말 됐는지"를 동시에 관리할 수 있다.

## Phase / Task 분해 원칙

- **Phase**: 의미 있는 마일스톤(예: 기반 구축 → 데이터 수집 → 콘텐츠/페이지 → 기능 → 완성도 검증). 선행이 필요한 것을 앞에 둔다.
- **Task**: Phase 아래의 실행 단위. **하나의 검증 가능한 산출물**로 끝나야 한다. 크면 쪼갠다.
- **의존성 우선 정렬**: DB 스키마·배포 설정 등 선행 Task를 먼저 배치.
- **초기 Phase에 배포 설정(config-as-code) Task**: 기반 구축 Phase에 **"배포 설정(config-as-code)" Task**를 둔다(Supabase 프로젝트·`vercel.ts`·환경변수·크론 스케줄). 상세 규약은 `web-deploy-config` 스킬을 참조. 다른 구현 Task가 여기에 의존하므로 앞 Phase에 배치한다.
  - ⚠️ **데이터 수집(크롤러·Python 배치·ETL) Task는 이 로드맵에 넣지 않는다** — 별도 모노레포 `wisdfire/jobhub-jobs` 소관이다. 대신 "수집 데이터가 Supabase에 적재돼 있다"를 **외부 의존성**으로 표기하고, 웹앱은 그것을 읽어 렌더링하는 Task만 둔다.
  - `docs/DATA-JOBS.md`가 있으면 그 문서를 가리키는 **외부 의존성 블록**을 로드맵에 둔다. 수집 적재를 기다리며 개발이 멈추지 않도록, 관련 Task는 **픽스처/시드 데이터로 개발·테스트 가능**하게 검증 기준을 잡고 실데이터 검증은 별도 항목으로 분리한다.
- **애드센스 Task 필수**: 애드센스 심사 요건(필수 페이지·콘텐츠 품질·기술 요건)을 충족하는 Task를 **반드시 포함**하고, 런칭 Phase의 완료 기준에 **애드센스 체크리스트 전항목 통과**를 건다 (`adsense-readiness` 스킬).
- **각 Phase에 테스트/QA Task 포함**: 모든 Phase에 해당 산출물을 검증하는 **테스트/QA Task**를 최소 1개 둔다(단위/통합/E2E, CI 게이트 구성 포함). 기능만 나열하고 테스트를 빠뜨리지 않는다.

## 체크박스 표기 규약 (필수)

모든 Task는 Markdown 체크박스로 진행 상태를 표시한다.

- `- [ ]` 미완료
- `- [x]` 완료

재호출·수정 시 **이미 `- [x]`인 Task는 보존**한다(진행 상태를 되돌리지 않는다).

## Task 표기 포맷

각 Task 한 줄에 다음을 붙인다.

```markdown
- [ ] T2.1 필수 정책 페이지 구현 — 의존성: T1.1 · 산출물: 개인정보/약관/문의/소개 4페이지 · 검증 기준: 4개 라우트 접근 가능 · PRD참조: §애드센스 준비
```

- **의존성**: 선행 Task ID(없으면 "없음").
- **산출물**: 이 Task가 남기는 구체적 결과물.
- **검증 기준**: 완료를 판정하는 객관 기준.
- **PRD참조**: 근거가 되는 PRD 요구사항.

## 검증 기준 연결 (Lighthouse 등)

- **완성도 Phase**에는 PRD의 Lighthouse 목표를 그대로 검증 기준으로 건다: `Performance / Accessibility / Best Practices / SEO 각 90+`.
- **각 Task 검증 기준에 테스트 통과 조건을 명시(필수)**: 기능 Task는 단위/통합/E2E 테스트 통과·접근성 위반 0 등 측정 가능한 기준을 건다. 예: `- [ ] T2.2 로그인 API 구현 — 검증 기준: 로그인 API 단위테스트 통과 · CI 게이트 green`.
- **CI 게이트**: 배포 관련 Task의 검증 기준에 "테스트 통과해야 배포"(CI 게이트 green)를 건다. PRD (d) 품질·테스트 전략과 짝을 맞춘다.
- PRD에 지표가 누락돼 있으면 기본 목표(Lighthouse 각 90+, 핵심 모듈 테스트 통과)를 제안하고 그 사실을 표시한다.

## 문서 구조

```markdown
## 진행 요약
- 완료 X / 전체 Y

## Phase 1: 기반 구축
- [ ] T1.1 배포 설정(config-as-code) — 산출물: `vercel.ts`·환경변수 · 검증 기준: 프리뷰 배포 성공·재현 가능 (참조: web-deploy-config 스킬)
- [ ] T1.2 ...
- [ ] T1.3 [QA] 기반 계층 테스트·CI 게이트 구성 — 검증 기준: 테스트 통과해야 배포(CI green)
## Phase N: 완성도 검증
- [ ] TN.1 Lighthouse 목표 달성 — 검증 기준: 4개 카테고리 각 90+
- [ ] TN.2 전체 테스트 통과율·커버리지 달성 — 검증 기준: 단위/통합/E2E 통과 · 커버리지 목표치 충족

## 리스크 · 선결 검증 필요
- (tech-verifier Blocker 항목은 정식 Task가 아니라 여기에)
```

## 원칙

- **PRD 추적성**: PRD에 없는 작업을 임의로 추가하지 않는다.
- **미검증은 Task 아님**: tech-verifier Blocker 항목은 "선결 검증 필요"에 둔다.
- **환각 금지**: 실현성 확인 안 된 기술을 완료 가능한 Task로 확정하지 않는다.
