# 스키마 설계 방법론

스키마의 필드명·PK는 파이프라인 전체(ETL·검증·모니터링)의 기준점이다. 여기서 흔들리면 하류 설계가 모두 어긋난다. **먼저 정확히 확정하라.**

## Why-first: 왜 3단계 모델링인가

소스에서 곧장 테이블로 뛰면 소스의 우연한 구조가 그대로 굳는다. 개념→논리→물리로 내려오면 "무엇을 저장하는가"와 "어떻게 저장하는가"를 분리해, 소스가 바뀌어도 버틸 모델을 얻는다.

1. **소스 분석**: 각 소스의 엔티티·필드·타입·카디널리티·갱신 주기·볼륨·유니크 식별자를 목록화한다. 샘플 데이터로 실제 분포·널 비율·이상값을 확인한다.
2. **개념 모델**: 핵심 엔티티와 관계(1:1/1:N/N:M)를 명사·동사로 도출한다. 저장 방식은 아직 정하지 않는다.
3. **논리 모델**: 엔티티를 릴레이션으로, 속성을 컬럼으로 매핑하고 후보 키·유니크 키를 식별한다.
4. **물리 모델**: PostgreSQL 타입·인덱스·제약·파티셔닝을 확정하고 Drizzle로 표현한다.

## 정규화 vs 반정규화 판단 기준

기본은 정규화(3NF)다 — 중복이 없으면 갱신 이상(update anomaly)이 없다. 반정규화는 **측정된 조회 병목**이 있을 때만 택하라.

- **정규화 유지**: 쓰기 빈번, 데이터 무결성 중요, 조인 비용 감당 가능.
- **반정규화 도입**: 읽기 압도적, 조인이 실제 병목, 집계/미리계산 컬럼이 반복 조회됨. 대신 갱신 일관성 유지 책임(트리거/ETL 재계산)을 명시하라.
- 판단할 때마다 근거(예상 쿼리 패턴, 읽기:쓰기 비율)를 문서에 남긴다.

## 타입 선택 (PostgreSQL 우선)

- 시각: `timestamptz`(UTC 저장) — 네이티브 로컬 타임 산술 대신 타임존 인식 타입.
- 금액·정밀 수치: `numeric` (부동소수 `float` 금지).
- 식별자: 자연키가 안정적이면 자연키, 아니면 `uuid`/`bigint` 대리키.
- 반구조 데이터: `jsonb` (검색·인덱싱 필요 시). 자주 쓰는 키는 별도 컬럼으로 승격.
- 열거값: `text` + CHECK 또는 lookup 테이블. enum 타입은 값 추가가 잦으면 피한다.

## 인덱스·제약 설계

- **인덱스는 조회 패턴에서 역산**: 예상되는 WHERE/JOIN/ORDER BY 컬럼에 인덱스. 추측으로 남발하지 말 것(쓰기 비용).
- 유니크·부분(partial)·복합 인덱스를 조합하고, 복합 인덱스는 선택도 높은 컬럼을 앞에 둔다.
- **제약을 스키마에 밀어넣어라**: NOT NULL, UNIQUE, CHECK, FK. DB가 강제하는 불변식은 검증 코드보다 강하다. 단, 검증 규칙과 이중 정의될 값은 validation 설계와 정합을 맞춘다.
- Upsert를 위한 유니크 키/PK를 반드시 명시한다(ETL의 ON CONFLICT 대상).

- 소스 분석 요약표
- 개념/논리 모델(엔티티·관계, 텍스트 다이어그램)
- 테이블별 컬럼표: 컬럼명 · 타입 · 제약 · 설명
- 관계·FK 목록, 인덱스 목록(+근거), 정규화 판단 근거
- **DDL 초안**과 **Drizzle 스키마 초안**(`pgTable`, 컬럼 헬퍼, relations) 병기
- PK/유니크 키를 굵게 표시(하류 참조 기준)

## 하지 말 것

- 존재하지 않는 PostgreSQL/Drizzle 기능 지어내기 → 불확실하면 Context7/공식 문서 확인.

- 조회 패턴 근거 없는 인덱스 남발, 근거 없는 반정규화.
