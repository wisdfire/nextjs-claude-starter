# QA Link Integrity — 링크 정합성 & 경계면 교차 검증

이 스킬은 QA의 **가장 중요한** 절차다. 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"**다. 두 쪽이 각각 올바르게 구현돼도 연결부에서 계약이 어긋나면 런타임에서 깨진다.

## 왜 정적 리뷰로는 못 잡나

- `fetchJson<T>()` 제네릭은 런타임 응답이 달라도 컴파일을 통과한다.
- `npm run build` 성공이 정상 동작을 보장하지 않는다(타입 캐스팅·`any`·제네릭 우회).
- "API가 있는가?"와 "API 응답이 호출측 기대와 일치하는가?"는 전혀 다른 검증이다.

그래서 **양쪽을 실제로 대조**해야 한다. 이 스킬의 스크립트가 그 대조를 기계적으로 자동화한다.

## 1. 링크/라우팅 정합성 (사용자 최우선 요구사항)

### 1-1. 파일 경로 → URL 추출 규칙

`app/` 하위 `page.tsx`(및 `route.ts`) 파일 경로에서 URL을 유도한다:

- **route group `(group)`은 URL에서 제거**한다: `app/(app)/dashboard/page.tsx` → `/dashboard`.
- **동적 세그먼트 `[param]`**은 파라미터 자리로 둔다: `app/posts/[id]/page.tsx` → `/posts/:id`.
- **catch-all `[...slug]`**은 와일드카드로 둔다.
- `app/page.tsx` → `/`. `app/api/**/route.ts` → `/api/...`(엔드포인트).

### 1-2. 링크 값 수집

코드 전체에서 다음을 수집한다:

- `href="..."`, `href={"..."}`, `<Link href="...">`
- `router.push("...")`, `router.replace("...")`
- `redirect("...")`, `permanentRedirect("...")`

### 1-3. 대조 알고리즘

각 링크 값에 대해:

1. 쿼리스트링(`?...`)·해시(`#...`)를 제거한다.
2. 외부 URL(`http://`, `https://`, `mailto:`)과 앵커(`#`)는 건너뛴다.
3. 동적 부분(`${...}`)은 파라미터 자리로 정규화한다.
4. 추출한 라우트 목록 중 **정확히 매칭되는 것이 있는지** 확인한다.
5. 매칭이 없으면 **404 유발 후보**로 보고한다.

### 1-4. route group 접두사 누락 집중 점검

가장 흔한 버그다. 예:

- 파일이 `app/(app)/dashboard/create/page.tsx`인데 링크가 `/create` → **404** (실제 URL은 `/dashboard/create`).
- 파일이 `app/(app)/projects/[id]/page.tsx`인데 링크가 `/projects/...` 인지 확인. group이 URL을 바꾸지는 않지만, 상위 세그먼트(`dashboard/` 등) 누락을 특히 의심한다.

정규화한 URL이 라우트와 "일부만" 겹치면(끝 세그먼트는 같은데 앞이 다르면) 접두사 누락으로 강하게 의심하고 리포트에 명시한다.

### 1-5. 자동화 스크립트 실행

반복 검증을 위해 번들 스크립트를 실행한다(프로젝트 루트에서):

```bash
node .claude/skills/qa-link-integrity/scripts/link-check.mjs
```

스크립트는 `app/`를 스캔해 라우트 목록을 만들고, 코드의 링크를 수집·대조하여 매칭되지 않는 링크를 파일:라인과 함께 출력한다. 종료 코드가 0이 아니면 문제가 있다는 뜻이다. 스크립트가 놓칠 수 있는 동적 경로는 수동으로 보완 확인한다.

## 2. API 응답 shape ↔ 프론트 훅 타입 교차 검증

**양쪽 동시 읽기**: `route.ts`의 `NextResponse.json()`과 대응 훅의 `fetchJson<T>`를 함께 연다.

검증 단계:

1. 각 API route의 `NextResponse.json(...)`에 넘기는 객체 shape을 추출한다.
2. 대응 훅의 `fetchJson<T>`(또는 fetch 후 사용 타입) `T`를 확인한다.
3. shape과 `T`가 일치하는지, **래핑 여부**가 맞는지 본다: API가 `{ items: [] }`를 주면 훅이 `.items`를 unwrap하는가? 배열을 기대하는 곳에 래퍼를 넘기면 `.map is not a function`.
4. **snake_case ↔ camelCase** 불일치를 본다: API `thumbnail_url` ↔ 타입 `thumbnailUrl`이면 값이 `undefined`가 된다.
5. **즉시 응답(202) vs 최종 결과**: 프론트가 즉시 응답에서 최종 결과 필드에 접근하지 않는지 본다.

## 3. API 엔드포인트 ↔ 프론트 훅 1:1 매핑 (호출 누락 탐지)

검증 단계:

1. `app/api/**/route.ts`에서 HTTP 메서드별 엔드포인트 목록을 추출한다.
2. 훅(`hooks/`·`lib/hooks/`)의 fetch 호출 URL 목록을 추출한다.
3. 엔드포인트 중 어떤 훅에서도 호출하지 않는 것을 식별한다 → "미사용" 플래그.
4. 미사용이 **의도적(관리 API 등)**인지 **호출 누락(기능이 동작 안 함)**인지 판단한다. `04_api_contract.md`의 "미사용" 표기와 대조한다.

## 4. 통합 정합성 체크리스트

```markdown
### 라우팅 정합성
- [ ] 코드 내 모든 href/Link/router.push/redirect 값이 실제 page 파일 경로(URL)와 매칭
- [ ] route group (group)이 URL에서 제거되는 것을 고려해 검증
- [ ] 동적 세그먼트 [id]가 올바른 파라미터로 채워짐
- [ ] route group 상위 접두사(예: /dashboard) 누락이 없음

### API ↔ 프론트 연결
- [ ] 모든 API 응답 shape과 대응 훅의 제네릭 타입이 일치
- [ ] 래핑된 응답({ items: [] })을 훅에서 unwrap
- [ ] snake_case ↔ camelCase 변환이 일관됨
- [ ] 즉시 응답(202)과 최종 결과 shape을 프론트가 구분
- [ ] 모든 API 엔드포인트에 대응 훅이 존재하고 실제 호출됨
```

## 5. 점진적 QA (incremental)

전체 완성 후 1회가 아니라 **각 모듈 완성 직후** 검증한다. 이유: 버그가 누적되면 수정 비용이 커지고, 초기 경계면 불일치가 후속 모듈로 전파된다. 백엔드 API 하나가 완성되면 즉시 그 API + 대응 훅을 교차 검증한다.

## 6. 발견 시 통보

발견 즉시 해당 에이전트에게 **파일:라인 + 원인 + 구체적 수정 방법**으로 통보한다:

- 링크 오류 → `frontend-engineer` (올바른 경로 제시).
- shape↔타입 불일치 → `frontend-engineer` **와** `backend-engineer` 양쪽(어느 쪽을 고칠지 합의 유도).
- 미호출 엔드포인트 → 양쪽에 통지.

리포트는 `_workspace/05_qa_report.md`에 통과/실패/미검증을 구분해 남기고, 링크 정합성 섹션을 최상단에 둔다.

## 하지 말 것

- "파일이 존재하는가"로 끝내지 않는다 — 반드시 "링크가 그 파일을 가리키는가"까지 본다.
- 한쪽 코드만 읽고 판단하지 않는다.
- 스크립트 결과만 믿지 않는다 — 동적 경로·조건부 렌더는 수동 보완한다.
- `npm run test` 출력을 읽지 않고 "통과"로 보고하지 않는다 — 새 테스트 하나가 아니라 전체 스위트를 돌린다(green 넘겨짚기 금지).
- 코드에 남은 `it.only`·`describe.only`(다른 테스트를 조용히 건너뜀)·`.skip`을 지나치지 않는다 — 미통과 항목으로 지적한다.
- 가짜 통과 테스트를 눈감지 않는다 — 검증 대상 자체 모킹·내부 상태 결합·고정 `sleep` flaky·기대값 수정으로만 덮은 테스트를 발견하면 리포트에 지적한다.
- 실패 테스트·빌드 경고를 임의로 "통과로 간주"하지 않는다 — 리포트에 그대로 남긴다.


## 애드센스 게이트 (차단성 · 필수)

> **`adsense-readiness` 스킬이 단일 진실 공급원**이다. 링크 정합성은 그 자체로 애드센스 요건이다 — 구글 프로그램 정책의 **Site behavior** 조항은 "사용자가 사이트를 쉽게 탐색해 약속된 정보에 도달할 수 있어야 한다"고 요구하므로, **깨진 링크(404)는 정책 위반 소지**다.

링크 검증에 더해 아래를 **Grep으로 결정적으로** 확인하고, 미충족은 **미통과(차단)**로 리포트한다:

- **개인정보처리방침 라우트 존재**(`app/privacy/**/page.tsx`)하고 구글 지정 문구(구글 광고 쿠키·opt-out 안내)를 포함하는가.
- **`app/robots.ts`가 `Mediapartners-Google`을 차단하지 않는가** (전역 `Disallow: /` 없음).
- **저가치 라우트(로그인·404/500·빈 목록·스켈레톤)에 광고 슬롯이 렌더되지 않는가** — 광고 컴포넌트 import를 라우트별로 대조.
- **플레이스홀더 화면이 없는가** — "준비 중", "Coming soon", "Under construction", TODO 스텁 Grep.
- **정책 페이지가 전역 내비게이션(푸터 등)에서 도달 가능한가.**
